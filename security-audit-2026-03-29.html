Findings & Remediations

Critical Fixed Secret key files world-readable (0644)

Impact: Any local user could read msg-signing-key.secret, db-encryption-key.secret, verifier-signing-key.secret, and iroh-key.secret — enabling message signature forgery, database decryption, credential minting, and QUIC endpoint impersonation.

Root cause: All four key files were created with std::fs::write() which uses the process umask (typically 0644 on Linux).

Files affected:

  • freeq-server/src/server.rs — load_msg_signing_key() + DB key creation
  • freeq-server/src/iroh.rs — load_or_create_secret_key()
  • freeq-server/src/verifiers/mod.rs — load_or_generate_signing_key()

Fix: New secrets module with write_secret() (Unix OpenOptions::mode(0o600)) and tighten_permissions() (detects and fixes overly permissive files on startup). All four creation sites use write_secret(); all four load sites call tighten_permissions().

976e690 — security: write secret files with 0600 permissions, tighten on load

High Fixed Auth broker HMAC replay protection was optional

Impact: If X-Broker-Timestamp was omitted, a captured signed request could be replayed indefinitely to mint web tokens or push PDS sessions. An attacker with a single captured request could impersonate any authenticated user.

Root cause: verify_broker_signature_raw() treated the timestamp header as optional "for backward compat". The HMAC covered only the body, not the timestamp.

Files affected:

  • freeq-server/src/web.rs — verify_broker_signature_raw()
  • freeq-auth-broker/src/main.rs — sign_body(), mint_web_token(), push_web_session_with_token()

Fix:

  • X-Broker-Timestamp is now required (rejects 401 if missing)
  • HMAC now covers ts={timestamp}\n || body (timestamp bound to signature)
  • Clock skew enforced at ≤60 seconds
  • Broker updated to always send timestamp and include it in MAC

12e34e5 — security: require X-Broker-Timestamp in HMAC, prevent replay attacks

High Fixed Web-token TTL mismatch (30min actual vs. 5min documented)

Impact: A stolen or intercepted web-token had a 30-minute window for misuse, six times longer than the documented 5-minute intent.

Root cause: Duration::from_secs(1800) in cap.rs while comments stated "5 minutes".

Files affected:

  • freeq-server/src/connection/cap.rs:166

Fix: Changed to Duration::from_secs(300) (5 minutes). Tokens remain single-use on first SASL success; broker issues fresh tokens on each /session call.

ceb5787 — security: reduce web-token TTL from 30 minutes to 5 minutes

Medium Fixed Secrets default to current working directory

Impact: Secret files could be written into the repository checkout or shared directories, risking accidental version-control commits or broader read access.

Root cause: data_dir() fell back to "." when neither --data-dir nor --db-path was set.

Fix: Now defaults to platform state directory:

  • Linux: $XDG_STATE_HOME/freeq or ~/.local/state/freeq
  • macOS: ~/Library/Application Support/freeq
  • Falls back to CWD only if no HOME is available, with a WARN log

a398d21 — security: default data_dir to platform state dir, warn on CWD fallback

Medium Fixed No HTTP rate limiting on REST proxy/upload endpoints

Impact: DoS potential via /api/v1/og (OG preview with outbound fetch), /api/v1/blob (PDS proxy), and /api/v1/upload (PDS fan-out) — even with existing SSRF defenses, the server could be used as an amplifier.

Root cause: No rate limiting middleware on REST routes.

Fix: Added per-IP sliding-window rate limiter (30 requests / 60 seconds) to all three expensive endpoints. Returns 429 Too Many Requests when exceeded.

8c8cf42 — security: add per-IP rate limiting on REST proxy/upload endpoints

Medium Fixed Crypto-at-rest silent plaintext fallback

Impact: On AES-GCM encryption failure, messages were stored in plaintext. On decryption failure, raw ciphertext was returned to users. Both silently degraded confidentiality without operator awareness.

Root cause: encrypt_at_rest() had Err(_) => plaintext.to_string(); decrypt_at_rest() had Err(_) => stored.to_string().

Fix:

  • Encryption failure now panics (indicates broken key — better to crash than store plaintext)
  • Decryption failure returns [decryption failed] placeholder and logs at ERROR
  • Legacy unencrypted data still returned as-is (logged at DEBUG for migration tracking)

a37360a — security: make encryption-at-rest failures hard errors, not silent fallback

Low Client SDK insecure TLS mode

tls_insecure: true in freeq-sdk completely disables TLS certificate verification, enabling MITM attacks. Default is false (safe). Recommend adding prominent doc warnings and considering a #[cfg(feature = "insecure-tls")] gate.

freeq-sdk/src/client.rs — rustls_insecure_config()

Low HSTS header sent regardless of TLS

Benign (browsers ignore HSTS over HTTP), but could be gated on TLS being enabled. No action needed.

freeq-server/src/web.rs — security_headers()

Positive Controls (Verified Secure)

Secure SSRF Defenses
  • Blob proxy: Strict host allowlist (PDS/CDN only) with Range support
  • OG preview: DNS resolution with private/RFC1918 rejection, DNS pinning via resolve(), 256KB body limit, HTML-only
Secure SASL Authentication
  • Nonce-bound PDS verification methods; cryptographic challenge-response via DID keys
  • Single-use challenges with HashMap removal (replay-safe)
  • Configurable challenge timeout (default 60s); DPoP nonce retry with 3-attempt cap
  • DID document ID mismatch check prevents cross-DID attacks
  • SASL failure counter with disconnect after 3 failures
Secure SQL Injection Prevention

All database queries use parameterized statements via rusqlite's params![] macro. No string concatenation for SQL.

Secure Web Security Headers

CORS restricted to known origins; CSP, X-Frame-Options (DENY), HSTS, Referrer-Policy, X-Content-Type-Options all present.

Secure Message Signing & Verification
  • Ed25519 per-session keypairs with canonical signing format (did\0target\0text\0timestamp)
  • MSGSIG registration requires authenticated DID (can't register for another DID)
  • Server fallback signing for clients without support
  • ULIDs use 80 bits of cryptographic randomness (unpredictable)
Secure Web Client XSS Prevention

No dangerouslySetInnerHTML, eval(), or innerHTML. React Markdown with skipHtml. URL sanitization with scheme whitelist. All links get rel="noopener noreferrer".

Secure OAuth / PKCE / DPoP

Full PKCE with SHA-256 code challenge. DPoP-bound tokens with nonce rotation. State TTL enforcement (5 min). Token file permissions 0600 in SDK.

Secure S2S Federation Transport
  • QUIC/TLS 1.3 via iroh with cryptographic peer identity
  • Allowlist enforcement on incoming connections
  • Trust levels (Full/Relay/Readonly) enforced per-event before state mutation
  • 100 events/sec per-peer rate limiting
  • S2S authorization checks on Mode, Kick, Ban, Topic, Join

Deployment Impact on Running Servers

This section describes what happens when these changes are deployed to servers currently running in production.

Immediate / Automatic Effects (No Action Required)

Secret file permissions (Critical fix): Existing .secret files will be automatically tightened from 0644 to 0600 on next server startup via tighten_permissions(). A WARN log will be emitted for each file tightened. No data loss. No service interruption.
Web-token TTL (High fix): Existing web-tokens older than 5 minutes will be rejected on next SASL attempt. Users with active sessions are unaffected (tokens are consumed on first use). Users with stale browser tabs (>5 minutes) will need to re-authenticate — the broker automatically issues fresh tokens on reconnect. Brief reconnect for some web users.
REST rate limiting (Medium fix): New per-IP rate limits activate immediately. Normal users will not be affected (30 req/60s is generous for link previews and uploads). Automated scrapers or abusive clients hitting these endpoints will start receiving 429 responses.
Crypto-at-rest error handling (Medium fix): No impact on existing data. Encryption now panics on failure (which should never happen with a valid key). Decryption failures on corrupted data will now return [decryption failed] instead of raw ciphertext — this is more correct behavior. Legacy unencrypted messages continue to be returned as-is.

Requires Coordination

Broker HMAC timestamp (High fix) — BREAKING CHANGE: The server now requires X-Broker-Timestamp on all broker requests, and the HMAC now covers ts={timestamp}\n || body.

The auth broker and freeq-server must be deployed simultaneously. If the broker is updated first (sends timestamp) but the server is old (ignores it), there is no problem — old server still accepts. But if the server is updated first (requires timestamp) and the broker is old (doesn't send it), all broker requests will be rejected with 401, breaking OAuth login for web users.

Recommended deployment order:
  1. Deploy the updated broker first (it now sends X-Broker-Timestamp)
  2. Verify broker health: GET /health
  3. Deploy the updated server
  4. Verify OAuth flow works end-to-end

Data Directory Change (Medium fix)

Default data directory (Medium fix) — BEHAVIOR CHANGE for new deployments: Servers started without --data-dir or --db-path will now write secrets to ~/.local/state/freeq/ instead of the current working directory.

Existing deployments with --data-dir or --db-path are completely unaffected.

If you rely on CWD for key storage (e.g., development), you'll need to either:
  • Add --data-dir . explicitly, or
  • Copy existing .secret files to ~/.local/state/freeq/
A WARN log is emitted if the CWD fallback is used (no HOME env var found).

No Impact

  • IRC protocol behavior: Unchanged. No client-visible protocol changes.
  • S2S federation: Unchanged. No wire protocol changes.
  • Message signing: Unchanged. Existing signed messages remain valid.
  • Database schema: Unchanged. No migrations needed.
  • TLS/SASL flow: Unchanged (the token TTL change only affects the acceptance window, not the protocol).

Critical Fixed REST API channel history exposed without authentication

Impact: /api/v1/channels/{name}/history returned full message history to any unauthenticated HTTP request, including for invite-only and key-protected channels.

Fix: Returns 403 Forbidden for channels with +i or +k modes. Public channels remain accessible (IRC model). Use IRC CHATHISTORY for restricted channels.

7bb6563

Critical Fixed Message edit/delete authorization used nick instead of DID

Impact: An attacker could change their nick to match an original message sender and edit or delete their messages. Critical integrity violation.

Fix: For authenticated users, authorship now verified by comparing DIDs via nick_owners. For guests, falls back to nick comparison (IRC compat). Ops can still delete any message in their channel.

5a4a6d0

Critical Fixed S2S message fields allow IRC protocol injection

Impact: A malicious federated peer could inject \r\n into S2S Privmsg from, target, or text fields, causing arbitrary IRC commands to be injected into local clients' streams. Also applied to Join nicks/channels and Topic text.

Fix: Added sanitize_s2s_str() that strips \r, \n, \0 and enforces length limits on all peer-provided fields before they reach IRC output.

1b0219e

High Fixed OAuth pending/complete maps had no TTL (memory exhaustion)

Impact: Attacker could spam OAuth flows to fill oauth_pending and oauth_complete HashMaps indefinitely, exhausting server memory.

Fix: Added 10-minute TTL with periodic pruning in the existing cleanup task. Added created_at timestamp to OAuthResult.

06bf978

High Fixed No channel name or topic length limits

Impact: Attacker could create channels with MB-length names or set MB-length topics, causing memory exhaustion and degraded performance.

Fix: Channel names limited to 64 characters, topics to 512 characters. Rejects with appropriate IRC error codes.

870e8d6

Verified Secure Nick ownership enforcement on NICK command

Initially flagged as a vulnerability, but code review confirmed the NICK handler at line 523-531 already checks nick_owners and rejects non-owners with "Nickname is registered to another identity". Registration at line 273-303 force-renames non-owners to Guest nicks. No fix needed.

Round 2 Deployment Impact

REST API history restriction: Channels with +i or +k will now return 403. Web clients using the REST history endpoint for private channels will need to use the WebSocket/IRC CHATHISTORY command instead. Public channel history is unaffected.
Edit/delete DID check: Authenticated users can only edit/delete messages sent by nicks they own (same DID). Guests can still edit/delete their own messages by nick. No behavioral change for users who don't change nicks.
S2S field sanitization: Transparent to legitimate peers. Only affects peers sending malformed strings with control characters. Length limits (512 for from, 200 for channel/target, 4096 for text) are generous and won't affect normal operation.
OAuth map cleanup: OAuth flows older than 10 minutes are now pruned. Users who start an OAuth flow and don't complete it within 10 minutes will need to restart. Normal OAuth flows complete in seconds.
Channel/topic limits: Channel names >64 chars and topics >512 chars are rejected. No known channels use names this long. Standard IRC limits are even shorter (50 chars).
