## Validation Criteria

- Returning the app to foreground while the existing IRC connection is alive does
  NOT trigger `fetchBrokerSession()`. Verified by adding a counter to the broker
  and observing it stays flat across foreground/background cycles when network
  is unchanged.
- Broker responding 502/503/504 for ≤ 30 s does NOT clear the saved broker token.
  Only a sustained, genuine 401 (3 in a row across reconnect attempts, plus
  beyond the 14-day grace window) clears credentials.
- `consecutive401Count` clears on every successful broker round-trip — verified
  in `fetchBrokerSession()` test path.

## Motivation

`AppState.handleScenePhase(.active)` checks `connectionState == .disconnected`
but does NOT consult transport liveness. After the WebSocket migration this
matters more, because WS connections survive brief background pauses better
than raw IRC over 6667 — but the app would still drop the live connection
and hit the broker on every wake.

The `consecutive401Count` + `canAutoClearBrokerCredentials` pair is structurally
correct (`AppState.swift:711, 720, 765`), but it has no diagnostic logging and
the 4-attempt-per-call backoff in `fetchBrokerSession()` already makes it
extremely sensitive to transient broker behaviour. We should add observability
and verify the discrimination is doing what we think.
