## Validation Criteria

- iOS app's first transport choice is `wss://irc.freeq.at/irc`, not `irc.freeq.at:6667`.
- A connect attempt that cannot reach the server completes in ≤ 10 s with a clear error,
  not an indefinite hang.
- When WebSocket connect fails, the SDK falls back to plain TCP within the same
  10 s budget.
- On a network that blocks port 6667 (verifiable via a simulated firewall or by
  toggling cellular hotspots that filter IRC), the iOS app reaches IRC registration
  via WebSocket without user intervention.

## Motivation

Today the Rust SDK only speaks raw TCP and `TcpStream::connect()` has no timeout
(`freeq-sdk/src/client.rs:680`). On any network where 6667 is blocked the iOS app
hangs in `.connecting` for the OS-default ~75 s, and the user falls off the
"Sign in manually" cliff at 15 s — losing their session for a transport-layer
problem that has nothing to do with auth. The web client already proves WebSocket
works against the exact same server endpoint (`freeq-server/src/web.rs:173`).
This is the single highest-leverage fix.
